## 🏗️ Proposed Architecture Changes

### API Key Management Strategy

**User Journey:**
1. User signs in via Google (Supabase OAuth)
2. On first access to presentation editor, modal prompts: "Enter your OpenAI API key to enable AI features"
3. User enters key → validated client-side (format check + minimal API test)
4. Key stored **encrypted at rest** in `user_profiles.openai_api_key_encrypted` (AES-256-GCM, key derived from user's password hash salt)
5. On each AI request, server decrypts user's key from DB and uses it for OpenAI API calls
6. Settings page allows key rotation, viewing masked key, and opting out

**Security:**
- Encryption key: derived from `user.id + password_hash_prefix` using PBKDF2 (or a server-side KEK stored in Vercel/Netlify env)
- Encrypted column in `users` table: `openai_api_key_encrypted TEXT`
- Audit log: record key rotations (timestamp only, no plaintext)
- Rate limiting: per-user + per-IP
- Never log actual keys

**Fallback:**
- If user has no key, show friendly UI explaining why key needed + link to obtain key
- Disable AI generation buttons; show upsell/tutorial CTA

### Phase 2: API Key Management Implementation (Day 1–2)
**Skill:** test-driven-development + subagent-driven-development (parallel tasks)

**2A — Database Schema Migration**
- Add `openai_api_key_encrypted` and `openai_api_key_iv` columns to `users` table
- Create migration: `prisma migrate dev --name add_encrypted_api_key`
- Add index for quick lookup

**2B — Encryption Utilities (New Module `src/lib/crypto/key-encryption.ts`)**
- `encryptApiKey(plaintext: string, userId: string): { encrypted: string; iv: string }`
- `decryptApiKey(encrypted: string, userId: string): string`
- Derive KEK from server-side env `API_KEY_ENCRYPTION_MASTER_KEY` + user ID salt
- Unit tests with test vectors (round-trip encryption/decryption)

**2C — API Routes for Key Management**
- `POST /api/user/api-key` — store encrypted key (server-side) — auth required
- `GET /api/user/api-key` — retrieve masked key for settings page
- `DELETE /api/user/api-key` — revoke/clear key
- All routes validate request origin, rate-limited

**2D — Modal Component (Client)**
- `src/components/settings/ApiKeyModal.tsx`
  - Triggers on first generation attempt if no key present
  - Input field with show/hide toggle
  - Validation: format `sk-...` + call `/api/validate-key?key=...` to check validity (without storing)
  - Save to localStorage **and** POST to server encrypted endpoint
  - Success confirmation + dismiss
  - Error states: invalid format, OpenAI API error, network failure

**2E — Settings Page**
- `src/app/settings/page.tsx` — user profile settings
- Section: "OpenAI API Key"
  - Show masked key (last 4 chars only)
  - Change key button (reopens modal)
  - Remove key button (with confirmation)
  - Toggle: "Use server-side encrypted storage" (default ON) — if OFF, client-only localStorage

**2F — Refactor AI Actions to Use Per-User Keys**
- Modify `src/app/_actions/image/generate.ts` and `src/app/_actions/presentation/generate/route.ts`:
  - Instead of reading `env.OPENAI_API_KEY`, call helper `getUserApiKey(userId)`:
    - If user has encrypted key in DB → decrypt + use
    - Else fall back to `env.OPENAI_API_KEY` (shared/default) but log warning
- Add error handling: "API key invalid or missing. Please update in Settings."

**2G — Tests**
- Unit tests for encryption/decryption (TDD)
- Unit tests for API route handlers (Supertest)
- E2E: "First-time user sets API key via modal, then generates presentation"
- E2E: "Settings page key management (rotate/revoke)"

**Deliverables:** Fully functional API key modal, encrypted server storage if user opts in, fallback to client storage, refactored AI actions.

## 📈 Risk Mitigation

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| DB connectivity remains blocked | Low | HIGH | Systematically debug network; use Supabase client fallback (connection pooling, retry logic) |
| API key compromise | Med | HIGH | Encrypt at rest, never log, rate-limit per user, monitor for abuse |
| Test coverage target not met | Low | MED | Prioritize testing critical paths first; use jest/playwright coverage collection |
| Encryption implementation errors | Low | HIGH | Use battle-tested libraries (Web Crypto API, `crypto-js`); get external security review |
| User confusion about API keys | Med | MED | Clear UX, tooltips, FAQ link, demo video |
| Performance degradation with per-user key decryption | Low | LOW | Cache decrypted keys in memory (short TTL), batch queries |
| Rate limits on OpenAI API from user keys | HIGH | MED | Document per-user rate limits; suggest batch generation strategy |
